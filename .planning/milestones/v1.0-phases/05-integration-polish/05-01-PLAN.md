**Backend — add experiment_count to Project model and list_projects:**

1. In `backend/models/project.py`, add `experiment_count: int = 0` as an optional field to the `Project` class (after `updated_at`). This field is computed, not stored — it defaults to 0 so single-project fetches still work without a join.

2. In `backend/services/project_service.py`, update `list_projects` to count experiments per project. After fetching projects, do a second query to count experiments grouped by project_id:

   ```python
   def list_projects(library_id: Optional[str] = None) -> list[Project]:
       query = get_client().table(_TABLE).select("*").order("created_at")
       if library_id:
           query = query.eq("library_id", library_id)
       result = query.execute()
       projects = [Project.model_validate(r) for r in result.data]
       if not projects:
           return projects

       # Count experiments per project
       project_ids = [p.id for p in projects]
       exp_result = get_client().table("experiments").select("project_id").in_("project_id", project_ids).execute()
       count_map: dict[str, int] = {}
       for row in exp_result.data:
           pid = row["project_id"]
           count_map[pid] = count_map.get(pid, 0) + 1
       for p in projects:
           p.experiment_count = count_map.get(p.id, 0)
       return projects
   ```

   Note: This fetches experiment rows to count client-side since Supabase Python client doesn't support `SELECT project_id, COUNT(*)...GROUP BY` directly. For the expected scale (tens of projects, hundreds of experiments) this is fine.

3. The router (`backend/routers/projects.py`) needs NO changes — it already serializes projects with `model_dump(by_alias=True)`, which will include `experimentCount` automatically.
