  <action>
**1. Install dependencies:**
```bash
cd frontend && npm install @dnd-kit/core @dnd-kit/sortable @dnd-kit/utilities
```

**2. Wrap RQSection tree in DndContext:**
   - Import `DndContext`, `closestCenter`, `DragOverlay`, `PointerSensor`, `useSensor`, `useSensors` from `@dnd-kit/core`.
   - Import `SortableContext`, `verticalListSortingStrategy`, `useSortable`, `arrayMove` from `@dnd-kit/sortable`.
   - Import `CSS` from `@dnd-kit/utilities`.
   - Configure PointerSensor with `activationConstraint: { distance: 5 }` to prevent accidental drags on click.

**3. Make each RQNode sortable:**
   - Use `useSortable({ id: rq.id })` hook in each RQNode.
   - Apply `transform` and `transition` to the node container via `style`.
   - Pass `listeners` to the drag handle icon (`drag_indicator`) only — NOT the entire row. This preserves click-to-edit on question text, status dropdown interaction, etc.
   - When `isDragging`, reduce opacity to 0.4.

**4. Implement onDragEnd handler in RQSection:**
   The handler must determine what kind of operation occurred based on the drop position:

   a. **Flatten the tree** into an ordered array with depth metadata for SortableContext.

   b. **Determine drop intent** from `active` and `over`:
      - Compute source (dragged item) and target (drop over item) from the flat list.
      - Use vertical position relative to target to determine: "above" (reorder before), "below" (reorder after), or "indent" (make child of target).
      - Simple approach for v1: treat all drops as sibling reorder within the same parent level. For reparenting, use a separate approach:

   c. **Reparenting via drop zones:**
      - At the root level (depth=0), if user drops between root items → reorder roots.
      - If user drops an item onto a different parent's children area → reparent to that parent.
      - For simplicity in v1: use the closestCenter collision strategy. When the drop `over` item is in a different parent group than the dragged item, treat it as a reparent operation. When in the same parent group, treat as reorder.

   d. **Constraint enforcement:**
      - Before executing reparent: check if dragged item has children (`rq.children.length > 0`). If yes and the operation would change its parent, abort the drop. Optionally log a message to console: "Cannot reparent an RQ that has sub-questions. Remove sub-questions first."
      - This enforces the locked decision: "Drag primary onto another primary = make sub-question (only if dragged item has no children)."

   e. **API calls after drop:**
      - If reparent: `researchQuestionsApi.update(draggedId, { parentId: newParentId || null, position: newIndex })`.
      - If reorder (same parent): update positions for all siblings in the affected level. Call `researchQuestionsApi.update` for each sibling with new position, OR use a batch reorder endpoint if available.
      - After API calls complete, call `onRefresh()` to re-fetch and rebuild the tree.

**5. DragOverlay:**
   - Show a simplified preview of the dragged RQ (question text + status badge) during drag. Use `DragOverlay` component from @dnd-kit.
   - Style with shadow-lg, bg-white, rounded-lg, px-3, py-2 for a floating card effect.

**6. Visual drop indicators:**
   - When dragging over a valid drop target, show a blue-400 border or line indicator.
   - When dragging over an invalid target (item with children being dropped onto a parent), show no indicator (or a subtle red indicator).

**Error handling:** If API calls fail during reorder/reparent, call onRefresh() to reset to server state. Log error to console.
